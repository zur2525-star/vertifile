Layer 1: File Integrity Hash

When you upload a document to Vertifile, the first thing that happens is the computation of a SHA-256 hash -- a unique cryptographic fingerprint of your entire file.

Why this matters: SHA-256 produces a 256-bit output that is practically impossible to reverse-engineer. If someone changes a single pixel, a single character, or a single byte of your document, the hash changes completely. There's no way to make a modified document produce the same hash as the original.

Think of it as the outer wall of a fortress. It won't stop a determined attacker alone -- but it's the foundation everything else builds on.

**Twitter/X:**
PVF Fortress -- Layer 2: HMAC Authentication

Now an attacker needs to crack both the hash AND the key.

Day 2 of 7. #PVFFortress #Vertifile

**LinkedIn:**
PVF Fortress -- Layer 2: HMAC Authentication

Yesterday we covered the file integrity hash. But here's the problem: if an attacker modifies your document, they could also recompute the hash for the modified version.

That's where HMAC (Hash-based Message Authentication Code) comes in.

HMAC combines the hash with a secret key that only Vertifile holds. The result is a signature that proves two things simultaneously:
1. The document hasn't been modified (integrity)
2. The signature was generated by Vertifile, not by an attacker (authenticity)

Without the secret key, an attacker cannot forge a valid HMAC -- even if they have the hash algorithm.

Two rings of the fortress secured. Five to go.
