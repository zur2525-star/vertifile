**Both** slots must be present. The primary is still the OLD key (good), and the next slot now holds the NEW key (good). The DB state hasn't flipped yet — the running app is still SIGNING with `$OLD_KEY` because DB `state='active'` still points at it.

**Failure modes in the Render boot log (Render dashboard → Logs):**
- `[key-manager] failed to parse ED25519_NEXT_PRIVATE_KEY_PEM` → bad PEM paste. Go to Chapter 2 Special, Rollback A.

- Service status "Deploy failed" or "Not live" → Rollback A.

**If the polling never shows `next=$NEW_KEY`:** the deploy isn't landing. Go to Chapter 2 Special.

## STEP 5 — Atomic DB flip (the actual rotation)

```bash
cd /Users/mac/Desktop/pvf-project

DATABASE_URL="$DATABASE_URL" node scripts/rotate-ed25519-key.js activate \
  --new-key-id="$NEW_KEY" \
  --grace-days=90 \
  --reason="scheduled-rotation-$(date +%Y%m%d)" \
  --health-url="$PROD_URL/api/health/deep"
```

**What this does:**
1. Pre-flight 1: verifies `$NEW_KEY` exists in `ed25519_keys` with `state='pending'`.
2. Pre-flight 2: verifies exactly ONE row has `state='active'` (the outgoing key).
3. Pre-flight 3: `GET $PROD_URL/api/health/deep` and confirms `$NEW_KEY` is in `ed25519_loaded_slots.primary` OR `.next`.
4. Opens one pg client, `BEGIN`, demotes old (`active → grace`) via `setEd25519KeyState`, promotes new (`pending → active`) via `setEd25519KeyState`, `UPDATE is_primary` flags, `UPDATE valid_until` on the old row to `NOW() + 90 days`, writes `key_rotation_log` row, `COMMIT`.
5. Invalidates the local `_activePrimaryCache` in this CLI process.

**Expected output (shape):**
```
Pre-flight: GET https://vertifile.com/api/health/deep
Pre-flight OK: new key loaded in slot=next

Rotation plan:
  outgoing key:  <OLD_KEY>  (active -> grace, retires 2026-07-08T...)
  incoming key:  <NEW_KEY>  (pending -> active)
  reason:        scheduled-rotation-...
  actor:         zur@...
  grace-days:    90

============================================================
  ROTATION COMMITTED — rotation_id=<N>
============================================================

Database state after rotation:
  <OLD_KEY>   state=grace,   valid_until=2026-07-08T...
  <NEW_KEY>   state=active  (current signing key)

IMPORTANT — 30 second signing tail
...
```

**Failure modes:**
- Any pre-flight fails → the transaction never opens. Read the error message, fix the cause, re-run.
- `Ed25519ForbiddenTransitionError` → the BEFORE-UPDATE trigger rejected a state change. This should be impossible in the happy path (`active→grace` and `pending→active` are both allowed). If you see it, the DB is in an unexpected state — do NOT retry blindly, `SELECT id, state FROM ed25519_keys` and investigate.
- Connectivity drop mid-transaction → the pg client's auto-rollback fires, the DB is unchanged. Re-run the `activate` command.

# CHAPTER 2 SPECIAL — RENDER BOOT FAILURE RECOVERY

You set the `ED25519_NEXT_*` env vars in STEP 3, Render redeployed, and the service is DOWN or returning errors. This chapter is the recovery playbook.

---

## Detecting the failure

Any one of these is sufficient evidence of a boot failure:

1. **Render dashboard shows "Deploy failed"** or the service status icon is red.
2. **Render logs** (dashboard → Logs) contain ANY of:
   - `[key-manager] failed to parse ED25519_NEXT_PRIVATE_KEY_PEM`
   - `[key-manager] ED25519_NEXT_KEY_ID must be exactly 16 lowercase hex characters`
   - `[key-manager] keyId env var does NOT match the keyId computed from the loaded PEM`
   - `[key-manager] ED25519_PRIMARY_KEY_ID and ED25519_NEXT_KEY_ID are identical`
   - `[key-manager] ED25519_REQUIRED=1 but no Ed25519 key loaded`
   - `PEM/keyId consistency check failed`
   - `Ed25519 signing key not loaded`
   - The process exited with a non-zero status code shortly after boot.
3. **`curl $PROD_URL/api/health/deep` returns 503 or times out** for >60s after the deploy timestamp.
4. **Upload smoke test** (from STEP 8) returns `ED25519_REQUIRED_NO_SIGNATURE`.

---

## Rollback Option A — remove the bad NEXT env vars (THE SAFE DEFAULT)

**Use this when:** the boot log shows a key-manager PEM/keyId parse error, the service is failing to boot, AND the DB still has `$OLD_KEY` as `state='active'` (i.e., you haven't run STEP 5 `activate` yet).

**Steps:**
1. Render dashboard → Environment.
2. **Delete** `ED25519_NEXT_PRIVATE_KEY_PEM`.
3. **Delete** `ED25519_NEXT_KEY_ID`.
4. **Save** → Render redeploys.
5. Wait ~60-90s for the redeploy.
6. Verify:
   ```bash
   curl -sS "$PROD_URL/api/health/deep" | jq '{status, primary_key_id, ed25519_signing_key_id, ed25519_loaded_slots}'
   ```
   **Expected:**
   ```json
   {
     "status": "online",
     "primary_key_id": "<OLD_KEY>",
     "ed25519_signing_key_id": "<OLD_KEY>",
     "ed25519_loaded_slots": { "primary": "<OLD_KEY>", "next": null }
   }
   ```

**Risk:** NONE. This is the untouch path. You're back to the pre-STEP-3 state: only the old key is loaded, the DB still says `$OLD_KEY` is active, and signing works with the old key.

**After Rollback A:** the DB still contains the pending `$NEW_KEY` row from STEP 1. Either retire it (if you're done with this rotation attempt) or keep it and try STEP 3 again with corrected env var values:
```bash
# Retire the pending key
DATABASE_URL="$DATABASE_URL" node /Users/mac/Desktop/pvf-project/scripts/rotate-ed25519-key.js retire-pending \
  --key-id="$NEW_KEY" \
  --reason="rollback-bad-render-boot-$(date +%Y%m%d)"
```

## Rollback Option B — DB state is bad

**Use this when:** the DB has a `state='pending'` row that the running app can't use (e.g., pending row was inserted but env var paste had a typo so the key won't load, OR you ran STEP 5 `activate` but STEP 8 smoke test fails with `ED25519_REQUIRED_NO_SIGNATURE` because the DB says `$NEW_KEY` is active but no instance has it loaded).

### Sub-case B1: pending key stuck, never activated

```bash
DATABASE_URL="$DATABASE_URL" node /Users/mac/Desktop/pvf-project/scripts/rotate-ed25519-key.js retire-pending \
  --key-id="$NEW_KEY" \
  --reason="stuck-pending-rollback-$(date +%Y%m%d)"
```

Then go to Rollback A to clean up any bad env vars.

### Sub-case B2: `activate` COMMITTED but fleet can't sign

**The cause:** DB says `state='active'` is `$NEW_KEY` but the running app processes don't have `$NEW_KEY` loaded in either slot (env var corruption, or deploy rolled back after DB commit).

**You CANNOT fix this by rolling the DB state back.** The BEFORE-UPDATE trigger forbids `grace → active` and `active → pending`. The rotation is permanent at the DB layer.

**The fix:** get the running app to load `$NEW_KEY`. Two paths:

**Path B2a (preferred):** re-set `ED25519_NEXT_PRIVATE_KEY_PEM` and `ED25519_NEXT_KEY_ID` in Render (same paste as STEP 3) and let it redeploy. Once the health endpoint shows `ed25519_loaded_slots.next = "$NEW_KEY"`, signing resumes (active-primary cache TTL = 30s, so signing comes back within 30s of the deploy landing).

**Path B2b (if B2a fails):** the private PEM for `$NEW_KEY` was lost (you cleared your terminal before pasting it into Render, and now the Render env var is gone). This is unrecoverable for the original `$NEW_KEY`. Run a NEW rotation:
```bash
# 1. Retire the CURRENT active key (which is the orphaned $NEW_KEY)
#    WAIT — the trigger forbids active→expired. You cannot retire active.
#    You MUST generate a NEW key and run activate to demote the orphaned one to grace.
# 2. Generate a brand new key:
DATABASE_URL="$DATABASE_URL" node /Users/mac/Desktop/pvf-project/scripts/rotate-ed25519-key.js generate \
  --reason="recovery-from-orphaned-$NEW_KEY"
# 3. Paste its env vars into Render (as STEP 3).
# 4. Wait for redeploy, then activate it (as STEP 5).
# 5. This demotes the orphaned $NEW_KEY from active→grace, which IS allowed.
# 6. Any PVFs signed by $NEW_KEY during the orphan window will verify against the grace row for 90 days.
```

Neon snapshot restoration (from Chapter 1.7) is the absolute last resort and is only acceptable if the orphan window is <5 minutes AND no new PVFs were issued against `$NEW_KEY`.

---

## NEVER DO THIS

- **NEVER `UPDATE ed25519_keys SET state='active' WHERE id='$OLD_KEY' AND state='grace'`.** The BEFORE-UPDATE trigger will block it with `Ed25519ForbiddenTransitionError`. Even if you could bypass the trigger (you can't from app code, but you could from a raw SQL console) — DON'T. It breaks the monotonic-forward invariant, corrupts `key_rotation_log`, and would let a compromised operator re-activate a revoked key. This is a permanent invariant locked in by Zur.
- **NEVER `DELETE FROM ed25519_keys WHERE state='active' OR state='grace'`.** Grace-state keys are still used for verification. Deleting them orphans every PVF signed during that window.
- **NEVER commit the private PEM to git.** Not to a temp branch, not to a gist, not to a `.gitignore`'d file you plan to delete later. It's in reflog forever.
- **NEVER share the private PEM in Slack, iMessage, email, Notion, Google Docs, or screenshots.** Use Render env var paste directly from terminal scrollback, then clear scrollback.
- **NEVER skip the 30s cache wait in STEP 6.** Polling during the window gets you mixed signals; acting on mixed signals during a rotation is how you corrupt state.
- **NEVER `--skip-preflight` on the `activate` command** unless you have independently verified (via Render logs, not health endpoint) that the running app loaded `$NEW_KEY`. The pre-flight is the ONLY automated check between "operator intent" and "DB state flip".
- **NEVER run `activate` against the wrong `DATABASE_URL`** (e.g., staging URL while you meant prod, or vice versa). The CLI reads `$DATABASE_URL` with no confirmation prompt. Double-check `echo $DATABASE_URL | head -c 50` before pressing enter.
